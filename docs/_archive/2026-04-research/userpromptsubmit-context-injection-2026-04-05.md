# Research: UserPromptSubmit Hook Context Injection

**Date:** 2026-04-05
**Scope:** Where the "Relevant Project Context" block injected on UserPromptSubmit events comes from, how commit data is sourced, and whether project-path filtering exists.

---

## Summary

The "Relevant Project Context" block is **not generated by claude-mpm source code**. It is generated by the external **kuzu-memory** CLI tool, registered as a UserPromptSubmit hook in `.claude/settings.local.json`. There is **no project-path filtering** — the hook uses a process-level `cwd`-based project root detection, and that project root is cached in a shared `/tmp` file that can cause cross-project contamination.

## Where the "Relevant Project Context" Block Is Generated

**File:**
`/Users/masa/.local/share/uv/tools/kuzu-memory/lib/python3.12/site-packages/kuzu_memory/cli/hooks_commands.py`

**Function:** `hooks_enhance()` (line 508)

**Exact block generation (lines 590–597):**

```python
elif memories:
    # Format as context
    enhancement_parts = ["# Relevant Project Context"]
    for mem in memories:
        enhancement_parts.append(f"\n- {mem.content}")

    enhancement = "\n".join(enhancement_parts)
    logger.info(f"Enhancement generated ({len(enhancement)} chars)")
    # Use hookSpecificOutput for silent context injection
    _exit_hook_with_json(context=enhancement)
```

The output format is JSON written to stdout:

```json
{
  "continue": true,
  "hookSpecificOutput": {
    "hookEventName": "UserPromptSubmit",
    "additionalContext": "# Relevant Project Context\n- <memory content>"
  }
}
```

Claude Code reads this and injects `additionalContext` into the prompt context as a system reminder.

## Where Commit Data Comes From

The memories injected as "Relevant Project Context" come from the **KuzuMemory graph database** stored in the project's `.kuzu-memory/memories.db` (or legacy `kuzu-memories/memories.db`).

Call chain:
1. `hooks_enhance()` → line 580: `_get_memories_with_lock(db_path, prompt, strategy="keyword")`
2. `_get_memories_with_lock()` (line 460) → `memory.attach_memories(prompt, max_memories=5, strategy=strategy)`
3. `MemoryService.attach_memories()` → delegates to `KuzuMemory.attach_memories()` (core/memory.py line 441)
4. `KuzuMemory.attach_memories()` → `RecallCoordinator.attach_memories()` (recall/coordinator.py line 172)
5. RecallCoordinator queries the KuzuDB graph with Jaccard keyword overlap scoring (`strategy="keyword"`)

Commit summaries stored in the database are populated by the `kuzu-memory hooks learn` PostToolUse hook and the `kuzu-memory hooks session-start` SessionStart hook. These hooks run git log commands internally (see `_git_sync_async()` at line 614 which runs `kuzu-memory git sync` as a subprocess with `cwd=str(project_root)`).

**Commit data is NOT read live from git log on each UserPromptSubmit.** It is retrieved from the pre-indexed KuzuDB graph database.

## Project-Path Filtering: Does It Exist?

**No.** There is no project-path filter applied during `attach_memories()` or in the RecallCoordinator query.

The filtering that exists is:
1. **Database isolation**: The hook resolves a `project_root`, derives `db_path = project_root / ".kuzu-memory/memories.db"`, and queries only that database. Memories from other projects are in different databases and are not queried.
2. **No field-level filter**: Once the correct DB is opened, all memories in it are eligible regardless of source path, session ID, or branch. The `user_id`, `session_id`, and `agent_id` parameters exist in the API but are **not passed** by `hooks_enhance()` (lines 580, 494–495).

## Critical Bug: Shared Project Root Cache

**File:** hooks_commands.py, lines 33–55 (`_get_cached_project_root`)

```python
def _get_cached_project_root() -> Path | None:
    cache_file = Path("/tmp/.kuzu_project_root_cache.json")
    try:
        if cache_file.exists():
            data = json.loads(cache_file.read_text())
            # 5 minute TTL
            if time.time() - data.get("timestamp", 0) < 300:
                cached_path = Path(data.get("path", ""))
                if cached_path.exists():
                    return cached_path
    except (json.JSONDecodeError, OSError):
        pass
    return None
```

This cache is a **single global `/tmp` file with a 5-minute TTL shared across all processes on the machine**. If a user switches between projects within 5 minutes, the second project's UserPromptSubmit hook will use the cached project root from the first project, and inject memories from the wrong database.

The `cwd` field from Claude Code's hook event (available in `input_data` as `input_data.get("cwd", "")`) is **read and discarded** — it is never passed to `find_project_root()`. Instead, `find_project_root()` is called with no argument, so it uses `Path.cwd()` (the process working directory), which may or may not match the Claude session's actual working directory.

---

## Where a Project Filter Should Be Added

To fix cross-project contamination, `hooks_enhance()` should:

1. Extract `cwd` from the hook event input:
   ```python
   cwd = input_data.get("cwd", "")
   ```

2. Pass it to `find_project_root()`:
   ```python
   project_root = find_project_root(start_path=Path(cwd)) if cwd else find_project_root()
   ```

3. Either disable the shared `/tmp` cache or make it keyed by `cwd`:
   ```python
   cache_key = hashlib.md5(str(cwd).encode()).hexdigest()
   cache_file = Path(f"/tmp/.kuzu_project_root_cache_{cache_key}.json")
   ```

The RecallCoordinator itself does not need a project-path filter because database isolation already scopes results to one project — the fix is entirely in `hooks_enhance()` to ensure the correct database is selected.

## Key Findings

1. **"Relevant Project Context" is generated by kuzu-memory** (`hooks_commands.py:590`), not by any claude-mpm source file.
2. **Data source is KuzuDB graph** — pre-indexed memories, not live git log.
3. **No project-path filter** in the recall query — all memories in the project DB are candidates.
4. **Database isolation** is the intended cross-project boundary, but it can be broken by the shared `/tmp` cache.
5. **The `cwd` field from the hook event is ignored** — project root detection uses `Path.cwd()` of the hook subprocess, which may not match the Claude session's working directory.
6. **Filter should be added** in `hooks_enhance()` by passing `cwd` from `input_data` to `find_project_root()` and keying the cache by cwd.
