## 1. Session Management Service

**Path**: `src/claude_mpm/services/session_management_service.py`

### WHAT

`SessionManagementService` (lines 35–314) is a `BaseService` + `SessionManagementInterface` subclass. It is the **orchestration layer** for launching Claude sessions; it does not directly spawn subprocesses but delegates to two session helper classes.

**Key responsibilities:**

- `run_interactive_session(initial_context)` (line 55): delegates to `InteractiveSession` (imported from `core.interactive_session`). Three-step flow: initialize → setup environment → handle interactive I/O.
- `run_oneshot_session(prompt, context)` (line 100): delegates to `OneshotSession` (imported from `core.oneshot_session`). Five-step flow: initialize → deploy agents → setup infrastructure → execute command → cleanup.
- `create_session_log_file()` (line 153): creates a timestamped `.jsonl` file under `.claude-mpm/logs/sessions/` (8-char UUID suffix).
- `log_session_event(log_file, event_data)` (line 185): appends a JSON line with UTC timestamp.
- `start_session(session_config)` (line 225): creates an in-memory session record under `self.active_sessions` dict (UUID key). Returns the UUID as session ID.
- `end_session(session_id)` (line 248): removes from `active_sessions`.
- `cleanup_sessions()` (line 295): async, purges sessions older than 24 hours.
- `get_service_status()` (line 209): returns dict with runner availability and active session count.

**Constructor** (line 38): accepts an optional `runner` (any object satisfying `ClaudeRunnerProtocol`, resolved at TYPE_CHECKING time to avoid circular imports). The `runner` is used by `run_oneshot_session` for `command_handler_service.is_mpm_command()`.

**Important**: `active_sessions` is an **in-memory dict only** — sessions are NOT persisted to disk and are lost if the service restarts.

### WHY

Docstring (lines 1–16) explains the service was extracted from `ClaudeRunner` to follow the Single Responsibility Principle. Protocol-based injection (`ClaudeRunnerProtocol`) avoids circular imports between the service layer and the core runner.

### DRIFT

- **CLAUDE.md is silent** on this service. CLAUDE.md's architecture overview (CLI → Services → Agents) implies services are intermediaries, which is consistent, but `SessionManagementService` is not named.
- **`active_sessions` dict is never persisted.** The `cleanup_sessions()` method exists and is async, but there is no scheduler that calls it — it is dead code in practice (no callers found in the codebase outside tests).
- **Session IDs generated by `start_session()`** (UUID v4) are completely separate from session IDs generated by `SessionManager` (the singleton in `services/session_manager.py`). Two independent ID namespaces coexist.

## 2. Session Manager (Singleton)

**Path**: `src/claude_mpm/services/session_manager.py`

### WHAT

`SessionManager` (lines 28–351) is a **thread-safe singleton** using double-checked locking (line 42–56). It provides a **single canonical session ID** for the entire process lifetime.

**Key behaviors:**

- `_generate_session_id()` (line 102): checks `CLAUDE_SESSION_ID`, `ANTHROPIC_SESSION_ID`, `SESSION_ID` env vars in order; falls back to a `%Y%m%d_%H%M%S` timestamp string (not a UUID).
- `get_session_id()` (line 125): thread-safe getter.
- `set_session_id(session_id)` (line 145): override with warning log.
- **Token tracking** (lines 165–243): `update_token_usage(input_tokens, output_tokens, stop_reason)` accumulates against a hardcoded `_total_budget = 200_000`. Updates `_context_metrics` dict. `should_warn_context_limit(threshold=0.70)` provides a threshold check.
- **Resume log integration** (lines 244–338): `_load_resume_log()` is called from `__init__` (line 100) — attempts to load a resume log from the previous session via `ResumeLogGenerator`. `generate_resume_log(session_state)` writes a resume log on demand.
- `reset()` (line 341): class method for test teardown only.

**Module-level globals**: `_manager: SessionManager | None = None` with `get_session_manager()` and `get_session_id()` convenience functions (lines 354–381).

**Default model hardcoded**: `"claude-sonnet-4.5"` appears in `_context_metrics["model"]` (line 87). This is stale — the codebase now uses `claude-sonnet-4-6` in active paths.

### WHY

Docstring (lines 1–15) states the service addresses "race conditions and duplicate session ID generation" by providing "a single source of truth for session identifiers."

### DRIFT

- **Two `SessionManager` classes exist** at different paths:
  - `src/claude_mpm/services/session_manager.py` — the singleton with token tracking (this file).
  - `src/claude_mpm/core/session_manager.py` — a completely different class, focused on subprocess reuse (creates sessions with UUID, persists to `~/.claude-mpm/sessions/` as JSON).
  - Both are named `SessionManager`. This creates significant confusion.
- **Hardcoded model name** `"claude-sonnet-4.5"` in `_context_metrics` (line 87) is stale. The codebase uses `claude-sonnet-4-6` elsewhere.
- **`_total_budget = 200_000`** (line 77) is hardcoded and never updated from actual Claude Code session data. The token tracking in this class is thus only approximate.
- **CLAUDE.md** does not mention this service or token tracking.

## 3. SDK Runtime (`SDKAgentRunner`)

**Path**: `src/claude_mpm/services/agents/sdk_runtime.py`

### WHAT

`SDKAgentRunner(AgentRuntime)` (lines 135–645) is the in-process Claude execution backend using the `claude-agent-sdk` Python package. The package is imported with a graceful fallback (lines 66–98): if `claude_agent_sdk` is not installed, `SDK_AVAILABLE = False` and sentinel `None` values are assigned so the module still imports cleanly.

**Data classes:**
- `ToolCallRecord` (line 106): captures a single tool invocation (name, input, output, approved, timestamp).
- `SDKAgentResult` (line 117): structured result with text, session_id, tool_calls, cost_usd, num_turns, duration_ms, is_error, raw_messages.

**Key execution modes:**
1. `run(prompt, config, **overrides)` (line 428): one-shot via `sdk_query()`. Collects all messages into a list then extracts result.
2. `run_with_hooks(prompt, tool_guard, blocked_tools, config)` (line 453): persistent session via `ClaudeSDKClient` with `can_use_tool` callback for tool interception. The `tool_guard` is an async `(tool_name, tool_input) -> bool` callable.
3. `inject(prompts, config)` (line 526): multi-turn: sends each prompt sequentially, waiting for `ResultMessage` before the next.
4. `resume(session_id, prompt, config)` (line 551): resumes a completed session via `options.resume=session_id`.
5. `fork(session_id, prompt, config)` (line 573): branches history via `options.resume=session_id, fork_session=True`.
6. `run_streaming(prompt, on_text, on_tool_call, config)` (line 597): fires async callbacks per `TextBlock`/`ToolUseBlock` as they arrive.
7. `interruptible(**overrides) -> InterruptibleSession` (line 643): returns a context manager wrapping `ClaudeSDKClient` with `interrupt()` support.

**Class constructors:**
- `__init__()` (line 149): raises `RuntimeError` immediately if `SDK_AVAILABLE` is False.
- `from_config(config: AgentConfig)` (line 178): creates from runtime-agnostic config; calls `resolve_model_to_sdk()`.
- `from_agent_template(template_path, mcp_servers)` (line 197): loads an MPM agent template JSON, extracts `narrative_fields.instructions` or `system_prompt`, model, allowed_tools, permission_mode, cwd, max_turns.

**Output style injection** (lines 277–315): `_build_options()` loads output style via `get_output_style_for_injection()` and prepends it to the system prompt. This compensates for SDK sessions not loading `~/.claude/settings.json` natively.

**`InterruptibleSession`** (lines 653–702): async context manager. `query(prompt)` sends prompt and collects response; `interrupt()` calls `client.interrupt()`; `session_id` property exposes last result session ID.

**Model name resolution** (from `agent_runtime.py` lines 63–106): `resolve_model_to_sdk("sonnet")` → `"claude-sonnet-4-6-20260124"`. The mapping table in `agent_runtime.py` line 64 shows `sonnet` maps to `"claude-sonnet-4-6-20260124"`.

### WHY

Docstring (lines 1–36) explains the module provides an alternative to subprocess-based execution for direct in-process communication. The `AgentRuntime` ABC makes higher-level code backend-agnostic. SDK_AVAILABLE graceful fallback allows the module to be imported in environments without `claude-agent-sdk`.

### DRIFT

- **Issue #523 referenced this as part of an "SDK Runtime epic" closed as "won't do."** However, `sdk_runtime.py` is a **fully implemented, non-stub** 746-line module. The epic was closed based on the prior belief that `_launch_sdk_mode()` did not exist, but as confirmed below, `_launch_sdk_mode()` exists in `core/interactive_session.py` (line 660) and is the actual entry point for interactive SDK mode.
- **Prior research doc** (`docs/research/sdk-runtime-current-state-2026-05-04.md` line 82) stated: "_`_launch_sdk_mode()` — DOES NOT EXIST_". This is **incorrect** as of the current codebase. The function is fully implemented at `src/claude_mpm/core/interactive_session.py` line 660.
- **`run_with_hooks()`** is not called from the primary interactive session path (`_launch_sdk_mode`). The interactive session uses `ClaudeSDKClient` directly with a `HookMatcher`-based hook config — it does not go through `SDKAgentRunner.run_with_hooks()`.

## 4. CLI Runtime (`CLIAgentRunner`)

**Path**: `src/claude_mpm/services/agents/cli_runtime.py`

### WHAT

`CLIAgentRunner(AgentRuntime)` (lines 28–230) wraps the `claude` CLI subprocess. It has no import of `ClaudeAdapter` — it builds `claude -p --output-format json` args directly.

**Key methods:**
- `_build_cli_args(prompt, resume_session, fork, output_json)` (line 69): assembles `["claude", "-p", "--output-format", "json", "--model", ..., "--max-turns", ..., "--resume", ..., "--fork-session", "--system-prompt", ..., prompt]`.
- `_invoke(prompt, resume_session, fork, config)` (line 100): uses `asyncio.create_subprocess_exec()`. Config overrides are applied temporarily (original values restored in `finally` block at line 138).
- `_parse_output(raw, duration_ms)` (line 155): tries JSON parse first; falls back to raw text. Extracts `result`/`text`, `session_id`, `cost_usd`, `num_turns`, `is_error`.
- `run(prompt, config)` (line 185): one-shot.
- `run_with_hooks(...)` (line 193): raises `NotImplementedError` — CLI subprocess does not expose tool interception.
- `resume(session_id, prompt, config)` (line 210): `--resume session_id`.
- `fork(session_id, prompt, config)` (line 220): `--resume session_id --fork-session`.

**Constructor** (line 36): accepts `system_prompt`, `model`, `cwd`, `max_turns`. Notably does NOT accept `allowed_tools`, `permission_mode`, `mcp_servers` (unlike `SDKAgentRunner`).

### WHY

Docstring: "Wraps the existing `ClaudeAdapter` subprocess execution to implement the `AgentRuntime` ABC." The comment is historically accurate but no longer literally true — the class does not import `ClaudeAdapter`. It was refactored to use `asyncio.create_subprocess_exec` directly.

### DRIFT

- **Docstring says "Wraps ClaudeAdapter"** but the implementation does not import `ClaudeAdapter`. The subprocess spawning is done natively via `asyncio.create_subprocess_exec`.
- **`CLIAgentRunner` has fewer config parameters than `SDKAgentRunner`**: no `allowed_tools`, `permission_mode`, `max_budget_usd`, `mcp_servers`. This makes the two runtimes not fully equivalent through `AgentConfig`.
- The CLI runtime is **not used** in the primary interactive session path. `_launch_sdk_mode()` uses `ClaudeSDKClient` directly; `_launch_exec_mode()` / `_launch_subprocess_mode()` use `os.execvpe` / PTY subprocess. `CLIAgentRunner` is used only via `runtime_bridge.execute_agent_prompt()` (sidecar/injection scenarios).

## 6. Runtime Bridge and Config

**Paths**:
- `src/claude_mpm/services/agents/runtime_bridge.py`
- `src/claude_mpm/services/agents/runtime_config.py`

### WHAT

**`runtime_config.get_runtime_type()`** (line 23): resolution order:
1. `CLAUDE_MPM_RUNTIME` env var (`"sdk"` or `"cli"`).
2. Auto-detect: `import claude_agent_sdk` → `"sdk"` if available, else `"cli"`.

**`runtime_config.get_runtime(config)`** (line 48): calls `get_runtime_type()` then `create_runtime()`.

**`runtime_bridge.execute_agent_prompt(prompt, ...)`** (line 17): unified entry point. If `session_id` provided → `runtime.resume(session_id, prompt)`; else → `runtime.run(prompt)`. Returns a dict with keys: `text`, `session_id`, `cost_usd`, `num_turns`, `duration_ms`, `is_error`, `tool_calls`, `runtime`.

### WHY

`runtime_bridge` provides a single dict-returning async function for callers (message endpoint, sidecar agents) that don't need the typed `AgentResult`.

### DRIFT

- **`runtime_config` does not read `configuration.yaml`** — the docstring (line 9) says "The config-file path (configuration.yaml) is intentionally omitted for now since the unified config system can layer this on top later." This is acknowledged in-code but not in any external doc.
- **The `--sdk` flag flow has a known gap**: `run.py` sets `CLAUDE_MPM_RUNTIME=sdk` via env var for the CLI flag, then `_setup_claude_runner()` reads it and passes `launch_method="sdk"` to `ClaudeRunner`. `ClaudeRunner.__init__` only accepts `"exec"` or `"subprocess"` (line 37 of `claude_runner.py`). However, `_launch_sdk_mode()` in `interactive_session.py` is invoked at line 237 when `self.runner.launch_method == "sdk"` — so the path DOES reach SDK mode. The `ClaudeRunner` constructor stores `launch_method` as-is without validation, meaning `"sdk"` is a valid runtime value in practice even though the docstring says "exec or subprocess."

## 7. Hook Event Bus

**Path**: `src/claude_mpm/services/agents/hook_event_bus.py`

### WHAT

`HookEventBus` (line 61) is a **file-based message queue** using a JSON-lines file at `~/.claude-mpm/hook_messages.jsonl` by default.

**`MessagePriority`** enum (line 20): `CRITICAL`, `HIGH`, `NORMAL`, `LOW`.

**`HookMessage`** dataclass (line 29): text, priority, source (e.g., `"monitor"`, `"ci"`), timestamp, metadata. Serializes/deserializes via `to_dict()` / `from_dict()`.

**`HookEventBus.send(message)`** (line 84): writes a JSON line using `fcntl.LOCK_EX` for exclusive file locking. Appends to the queue file.

**`HookEventBus.consume()`** (line 97): reads all lines, truncates file (exclusive lock), parses, sorts by priority (CRITICAL=0, HIGH=1, NORMAL=2, LOW=3). Returns `list[HookMessage]`.

**`HookEventBus.consume_for_hook()`** (line 143): called by PreToolUse hook. Increments `_tool_call_count`. LOW-priority messages are deferred (re-queued) unless `tool_call_count % 5 == 0`. Returns a formatted string for `systemMessage` injection, or `None` if nothing to inject.

**File locking**: uses POSIX `fcntl.flock()` — will not work on Windows.

### WHY

Docstring (line 1): "Sidecar agents write messages to a JSON-lines file. The PreToolUse hook reads and consumes them on each tool call." Design rationale is explicit: file-based queuing allows processes with no shared memory (sidecar agents, monitor agent background threads) to communicate across process boundaries.

### DRIFT

- **`HookEventBus` and `EventBus` (pyee) are separate systems** serving different purposes:
  - `HookEventBus`: file-based, for cross-process injection into Claude Code sessions via hook `systemMessage`.
  - `EventBus`: in-process pyee emitter, for decoupling hook handlers from Socket.IO relay (dashboard streaming).
  - These are not connected to each other.
- **`fcntl`-only locking** means this component is not portable to Windows environments. No fallback is provided.
- **The monitoring dashboard** does not read from `HookEventBus`. Dashboard events flow through `EventBus` → `DirectSocketIORelay` → Socket.IO. `HookEventBus` is solely the PM-injection channel.

## 8. Event Bus (pyee-based)

**Path**: `src/claude_mpm/services/event_bus/event_bus.py`

### WHAT

`EventBus` (line 27) is a **singleton** backed by `pyee.asyncio.AsyncIOEventEmitter`. It supports both sync publishing and async/sync handlers.

**Key fields:**
- `_enabled` (line 57): global on/off switch.
- `_event_filters` (line 58): set of allowed patterns; if empty, all events pass.
- `_event_history` (line 68): `deque(maxlen=100)` for debugging.
- `_handler_tasks` (line 73): set of `asyncio.Task` instances tracked to prevent GC.
- `_handler_wrappers` (line 76): dict mapping `(event_type, handler) -> wrapped_handler` for clean removal.
- `_wildcard_handlers` (lazily created in `on()`): dict of `prefix -> [handlers]` for wildcard subscriptions.

**`publish(event_type, data)`** (line 163): sync, thread-safe. Checks `_enabled`, filters, records in history, emits via pyee, also walks `_wildcard_handlers`. Updates stats.

**`on(event_type, handler)`** (line 254): if `event_type` ends with `"*"`, stores in `_wildcard_handlers` by prefix; otherwise wraps in a try/except safe_handler and registers with pyee.

**`publish_async(event_type, data)`** (line 241): simply delegates to `publish()` — pyee handles both.

**Filter logic** (line 137): empty filter set = pass all. Patterns ending with `"*"` match by prefix.

**Stats** (line 358): `events_published`, `events_filtered`, `events_failed`, `last_event_time`.

**Config** (`src/claude_mpm/services/event_bus/config.py`): `EventBusConfig` dataclass, all fields from env vars. Notable: `relay_enabled` defaults to `False` (`CLAUDE_MPM_RELAY_ENABLED=false`). Comment in config (line 52): "DirectSocketIORelay disabled by default — events already emit via direct `sio.emit()`."

### WHY

Module docstring: "Decouples hooks from Socket.IO implementation details. Allows multiple consumers for the same events. Enables easy testing without Socket.IO dependencies."

Singleton docstring: "Ensures single point of event coordination. Prevents duplicate event processing. Thread-safe initialization with proper locking."

### DRIFT

- **Two Socket.IO relay implementations coexist**:
  - `SocketIORelay` (`relay.py`): creates a Socket.IO *client* that connects back to localhost — acknowledged in `DirectSocketIORelay` as causing "events to not reach the dashboard properly."
  - `DirectSocketIORelay` (`direct_relay.py`): uses the server's internal broadcaster directly, bypassing client loopback.
  - The config comments (line 52 of `config.py`) confirm `SocketIORelay` is disabled by default because `DirectSocketIORelay` is the active path. `SocketIORelay` is therefore legacy/dead code in the default configuration.
- **Wildcard handler async scheduling** (lines 203–219) uses `asyncio.get_event_loop()` which raises a deprecation warning in Python 3.10+. The correct pattern is `asyncio.get_running_loop()` or `asyncio.get_event_loop_policy().get_event_loop()`.
- **`remove_all_listeners()`** does not remove wildcard handlers (line 330). Only pyee-registered handlers are cleaned up. Wildcard handlers in `_wildcard_handlers` accumulate permanently unless the singleton is reset.
- **CLAUDE.md** does not mention either `EventBus` or `HookEventBus`.

## 9. Monitor Agent

**Path**: `src/claude_mpm/services/agents/monitor_agent.py`

### WHAT

`MonitorAgent` (line 54) is a **background watchdog daemon thread** that polls session health and injects warnings into the PM via `HookEventBus`.

**`MonitorConfig`** dataclass (line 25):
- `token_limit = 200_000` (matches `SessionManager`).
- `warn_thresholds = [70, 80, 90, 95]` percent levels.
- `poll_interval = 10.0` seconds.
- `max_session_duration = 3600` seconds.
- `duration_warn_at = 0.8` (warn at 80% of max).
- `idle_timeout = 300` seconds.
- `auto_pause_threshold = 95` (triggers CRITICAL + auto-postmortem).
- Tool pattern thresholds: `consecutive_bash=5`, `consecutive_read=3`, `consecutive_write=3`, `consecutive_direct=8`.

**Lifecycle:**
- `start()` (line 82): spawns a daemon thread named `"mpm-monitor-agent"`.
- `stop()` (line 99): calls `_trigger_postmortem(reason="stop")` before setting stop event. Joins thread with 5-second timeout.
- `is_running` property (line 117): checks `_thread.is_alive()`.

**Health checks** (called every `poll_interval` from `_run_loop`):
1. `_check_context_pressure(state)` (line 167): reads `state["context_usage"]["tokens_used"]`, fires per-threshold warnings. At `auto_pause_threshold` (95%), sends CRITICAL and triggers `_trigger_postmortem(reason="critical_context")`.
2. `_check_session_duration(state)` (line 202): fires once when uptime ≥ 80% of `max_session_duration`.
3. `_check_idle_too_long(state)` (line 218): fires if state is `"processing"` and last activity is > 300s ago.
4. `_check_tool_call_patterns()` (line 253): walks last 50 activity events in reverse. Categorizes tools as implementation (Write/Edit), investigation (Read/Grep/Glob), execution (Bash), delegation (Agent/Task/TaskCreate/TaskUpdate), passive (TodoWrite/TaskList/etc.). MCP vector-search tools (`mcp__mcp-vector-search__*`) count as investigation. Tracks streaks with peak values across category switches. Fires circuit-breaker warnings labeled "PM CIRCUIT BREAKER #2."
5. `_check_per_query_limits(events)` (line 402): finds the most recent `user_input` event as a query boundary, counts investigation/read/vector-search tools since that boundary. Thresholds: `investigation_count >= 4`, `read_count >= 3`, `vector_search_count >= 2`.
6. `_check_incoming_messages()` (line 509): bridges `/mpm-message` tasks (via `TaskInjector`) to the PM session via `HookEventBus`. Deduplicates by task ID.

**Auto-postmortem** (lines 570–663): `_trigger_postmortem(reason)` is idempotent per reason (tracked in `_postmortem_triggered`). Spawns a daemon thread calling `_run_postmortem()`. Saves markdown reports to `~/.claude-mpm/postmortems/` with format `{timestamp}-{session_id}-{reason}.md`.

**Message injection** (line 677): `_inject_message(text, priority)` uses `HookEventBus.send()` with `source="monitor"`.

**Tool categories** (class variables lines 242–251):
- `_DELEGATION_TOOLS`: `{"Agent", "Task", "TaskCreate", "TaskUpdate"}`
- `_IMPLEMENTATION_TOOLS`: `{"Write", "Edit", "NotebookEdit"}`
- `_INVESTIGATION_TOOLS`: `{"Read", "Grep", "Glob"}`
- `_EXECUTION_TOOLS`: `{"Bash"}`
- `_PASSIVE_TOOLS`: `{"TodoWrite", "TaskList", "TaskGet", "ToolSearch", "AskUserQuestion"}`
- `_MCP_VECTOR_SEARCH_PREFIX`: `"mcp__mcp-vector-search__"`
- `_MCP_KUZU_RECALL_TOOL`: `"mcp__kuzu-memory__kuzu_recall"`

### WHY

Docstring (line 1): "Session watchdog for SDK mode. Runs as a background thread alongside the PM. Monitors session health and injects warnings via the hook event bus when thresholds are crossed." Comment at line 60: "The monitor is created and owned by `_launch_sdk_mode()` — it is not a global singleton."

### DRIFT

## 10. `_launch_sdk_mode()` — Interactive SDK Session Entry Point

**Path**: `src/claude_mpm/core/interactive_session.py`, line 660

### WHAT

`_launch_sdk_mode()` (line 660) is an instance method of `InteractiveSession`. It runs an asyncio event loop (`asyncio.run(_run_sdk_session())`) that:

1. Imports `claude_agent_sdk` — exits with error message if not available.
2. Builds system prompt from `self.runner._create_system_prompt()`.
3. Detects GitHub context and injects it into the system prompt (best-effort, timeout 3s).
4. Sets up `HookEventBus` and a `PreToolUse` hook via `hook_factory.create_pretooluse_hook()`.
5. Selects model: `CLAUDE_MPM_PM_MODEL` env > user's `~/.claude/settings.json` `"model"` field > `"claude-sonnet-4-6"`.
6. Sets `CLAUDE_CODE_SUBAGENT_MODEL=claude-sonnet-4-6` if not already set.
7. Creates `ClaudeAgentOptions` with system_prompt, cwd, permission_mode, hooks config.
8. Initializes `SessionStateTracker`, sets it as the global tracker.
9. Parses `--resume` from `runner.claude_args` if present.
10. Starts `MonitorAgent` (best-effort, failure is a no-op).
11. Starts `SDKEventBridge` (best-effort, failure disables dashboard streaming).
12. Enters `async with ClaudeSDKClient(options=options) as client:` main loop.
13. Creates `SDKCommandRouter(client=client, tracker=tracker)` for slash commands.
14. Input loop: reads from stdin, routes `/command` strings through `command_router`, sends other input via `client.query(user_input)`, processes response messages.
15. Response processing: handles `AssistantMessage` (TextBlock → print + tracker, ToolUseBlock → tracker + agent tracking), `ResultMessage` (tracker.record_result), `SystemMessage` (display).
16. On exit: calls `monitor.stop()` if monitor was started.

**State machine transitions via tracker:**
- `user_input` → `record_user_input()` → state = PROCESSING.
- `ToolUseBlock` → `record_tool_call(tool_name)` → state = TOOL_CALL.
- `ResultMessage` → `record_result()` → state = IDLE.
- On exit → `record_stopped()` → state = STOPPED.

### WHY

The function exists because `os.execvpe` (exec mode) replaces the MPM process entirely, preventing any post-Claude cleanup, monitoring, or injection. SDK mode keeps the MPM process alive, enabling the full monitoring/injection stack.

### DRIFT

- **Prior research doc** (`sdk-runtime-current-state-2026-05-04.md` line 82) stated `_launch_sdk_mode()` "DOES NOT EXIST." As of this investigation, the function is fully implemented at `interactive_session.py` line 660. The prior research may have been conducted on an older commit.
- **`channels_arg` branch** (line 668): if `runner.channels` or `CLAUDE_MPM_CHANNELS` is set, `_launch_sdk_mode()` delegates to `_launch_channel_hub_mode()` instead. This is an undocumented fork of the SDK interactive flow.
- **`ClaudeRunner` docstring** says it only accepts `"exec"` or `"subprocess"` for `launch_method`, but it stores `"sdk"` correctly via `__init__`. The fork at line 236 (`if self.runner.launch_method == "sdk":`) is fully wired.

## 11. Session State Tracker

**Path**: `src/claude_mpm/services/agents/session_state_tracker.py`

### WHAT

`SessionStateTracker` (line 56): thread-safe state store for the SDK interactive session.

**`SessionState`** enum (line 17): `IDLE`, `PROCESSING`, `TOOL_CALL`, `STARTING`, `STOPPED`.

**Write methods** (from REPL asyncio thread): `set_state()`, `set_session_id()`, `set_model()`, `record_user_input()`, `record_tool_call()`, `record_tool_result()`, `record_assistant_message()`, `record_result()`, `record_stopped()`.

**Read methods** (from HTTP server thread): `get_session_state() -> dict` and `get_activity(limit) -> list[dict]`.

**Activity buffer**: `deque(maxlen=100)` of `ActivityEvent` records. Each `ActivityEvent` has type, timestamp, preview (first 200 chars), tool name, status (`"running"` / `"complete"` / `"error"`), metadata.

**Module-level singleton** (lines 216–229): `_global_tracker: SessionStateTracker | None = None` with `get_global_tracker()` and `set_global_tracker()`. Set by `_launch_sdk_mode()` (line 858) at session start.

### WHY

Docstring: "Updated by `_launch_sdk_mode()` as it processes messages. Read by `MessageEndpoint` HTTP handlers (`/session`, `/activity`)." The tracker decouples the REPL asyncio thread from the HTTP monitoring thread without locking on every message.

### DRIFT

- **Tracker is only populated in SDK mode.** In exec/subprocess mode, `get_global_tracker()` returns `None`. Any code that calls `get_global_tracker()` without checking for `None` (e.g., `MonitorAgent._check_session_health()` line 148 handles this correctly with an early return) would fail silently.
- **Token accumulation in `record_assistant_message()` and `record_result()`** both add `input_tokens + output_tokens`. If a `ResultMessage` already includes the tokens from the preceding `AssistantMessage`, tokens may be double-counted.

## Summary: Most Significant Drift Findings

1. **`_launch_sdk_mode()` was previously documented as non-existent but is fully implemented** (717 lines of interactive SDK session logic) at `src/claude_mpm/core/interactive_session.py` line 660. The SDK Runtime epic being closed "won't do" appears to be based on stale information. `MonitorAgent` is instantiated and running in this path.

2. **Two independent `SessionManager` classes** coexist with identical names at different paths (`services/session_manager.py` singleton with token tracking, `core/session_manager.py` with subprocess-reuse/UUID persistence). Neither is referenced in CLAUDE.md. The singleton's token tracking uses a hardcoded stale model name (`claude-sonnet-4.5`).

3. **Two Socket.IO relay implementations coexist** (`SocketIORelay` via client loopback vs. `DirectSocketIORelay` via server broadcaster). `SocketIORelay` is acknowledged as broken in `direct_relay.py`'s docstring and is disabled by default. It is retained in the codebase but is dead code in the default configuration.

4. **`CLIAgentRunner` docstring falsely claims it wraps `ClaudeAdapter`** — it does not import `ClaudeAdapter` and spawns subprocesses natively via `asyncio.create_subprocess_exec`. The CLI runtime also lacks `allowed_tools`, `permission_mode`, `max_budget_usd`, `mcp_servers` support, making it not feature-equivalent to `SDKAgentRunner`.

5. **`SessionManagementService.cleanup_sessions()`** (the async cleanup method that purges sessions older than 24 hours) has no callers in production code — it is invoked only in tests. The `active_sessions` dict it manages is also in-memory-only, so the cleanup has no persistent effect anyway.
